全类型标准增量数据入库与状态治理方案 (V3 优化版)
基于前期的测试反馈与业务深挖，本方案旨在对现有的数据入库逻辑进行纠偏，并引入一套高度自动化的“标准生命周期状态引擎”。同时，本规范不仅适用于国家标准（GB），也将作为通用准则，指导行标、地标、团标及国际标准的全面接入。

1. 核心入库逻辑的四大修正策略
   1.1 单位画像优化：废止截断，保留实体全称
   痛点：原逻辑对起草单位进行清洗时截断了“有限公司”等后缀，导致企事业单位名称失真。
   修正方案：全面弃用名称后缀正则过滤逻辑。对于任何标准类型提取出的起草单位，仅执行首尾去空格操作，100%保留其原始全称。在此基础上进行查重，并建立主表与 unit_dict 的关联。
   1.2 替代关系修复：重构分隔符体系
   痛点：此前的代码利用空格（\s）拆分“被替代标准”字段，导致诸如 GB 11557-2011 这类内部自带空格的标准号被意外截断，致使 std_replace 表无法准确找到对应老标准并追加数据。
   修正方案：重构切分规则，强制只使用中英文逗号、分号及顿号 [，,；;、] 作为切割依据。
   1.3 谱系族传承与置顶机制
   当新数据替代了老数据时，为保障谱系演化图的精准渲染，执行以下操作：

PID 自动继承：若被替代的老标准已被编入某一 ped_id 家族，新标准直接沿用该 PID（不再随机生成新家族）。
家族代表易主：自动将该 ped_id 对应的 std_id_latest 字段更新为当期导入的新标准号。
链条排序置顶：在生成拓扑图 ped_chain 的 JSON 时，通过数组插入逻辑，确保新标准名称永远出现在 nodes 列表的第一位（索引 0）。
1.4 历史兼容：并行记录冗余表
在将起草单位精准拆解入现代画像表的同时，必须继续将包含所有单位的原始长字符串整体插入到 std_extend_h 表的 draft_unit 字段中，确保与老系统的纵向兼容。

2. 状态自动化级联与定时治理机制 (Automated Governance)
   标准的“现行”与“废止”并非静态数据，必须建立以亚洲/上海 (Asia/Shanghai) 为时区基准的动态引擎。

2.1 动作一：数据导入时的即时级联
新标准为“现行”：在入库成功的那一刻，系统立即根据替代关系顺藤摸瓜，将被替代的旧标准在 std_base 中的 ex_state 修改为 “废止”。
新标准为“即将实施”：系统按兵不动，被替代的旧标准暂时继续维持“现行”状态。
2.2 动作二：时间轴自动治理任务 (CRON Job / Post-Import Trigger)
触发机制：
系统后台每隔 30天 自动运行一次。
每次管理员通过 Web 界面导入新数据完成后，立刻主动唤醒运行一次。
执行逻辑：
获取当前上海时间的零点日期。
扫描 std_base，找出所有 ex_state = 2 (即将实施) 且 implement_date <= 当前日期 的标准。
转正：将这些逾期或恰逢当日的标准的 ex_state 更新为 1 (现行)。
追溯作废：紧接着，查找这些刚刚转正的标准所替代的所有前身，将其状态强制变更为 0 (废止)。3. 全类别扩展指南 (HB / DB / TB / ISO / IEC)
上述所有的修正与自动化逻辑，底层均依赖于主干表 std_base 与关系表 std_replace、std_pedigree。因此，当我们扩展数据源到其他标准时，该体系天然兼容。在具体的开发实施中，需注意以下针对性适配：

3.1 跨维度的谱系穿透
在实际演化中存在“行标升国标”或“地标转行标”的现象（例如 JB/T 被升级为 GB/T）。

实施规范：替代关系和谱系计算模块严格不区分标准前缀。凭借全库唯一的 base_id，系统应当允许且能够完美绘制跨标准类别（跨域）的谱系树。
3.2 详情字段的精准分流
针对导入的其它标准类型数据，依据 std_type_no 动态路由：

01 行标 / 02 地标：需着重解析并落实 industry_type（行业领域）、record_no（备案号）、approve_dept（批准发布部门）。
03 团标：确保 gbc（国民经济分类）准确无误地落入 std_tb_detail。
04/05 国际标准：在解析时抓取并存入特定的版本标识（如 std_version: 4.1）。
3.3 翻译接口的差异化屏蔽
我们在 web_import_tool.py 中内置了百度 API 用于自动补全空缺的英文名。

实施规范：对于 ISO、IEC 等国际标准，因其原始语种具备高权威性，应在代码中屏蔽此功能（若 std_type_no 属于国际标准类型，无论英文名是否为空，均不触发机器翻译），防范语义偏差。
